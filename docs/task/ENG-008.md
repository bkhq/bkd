# ENG-008 ACP 前端改造为协议原生时间线

- **status**: completed
- **priority**: P1
- **owner**: codex
- **plan**: PLAN-017
- **created**: 2026-03-13

## 描述

当前前端仍以既有 Claude/Codex 风格日志为中心，再把 ACP 事件映射为 `system-message`、`tool-use`、`task-plan` 等旧语义。随着 ACP agent 增加，这种兼容层会持续膨胀，维护成本越来越高。

本任务目标是先完成 ACP-native 前端时间线的调查与方案设计，明确 view model、渲染层边界、兼容策略和最小落地路径。

## 调查结果

- 当前前端主链路是 `useIssueStream() -> useChatMessages() -> SessionMessages`，数据核心仍是 `NormalizedLogEntry[]`。
- [apps/frontend/src/hooks/use-chat-messages.ts](/app/bkd/worktrees/n6ggd95o/l2801201/apps/frontend/src/hooks/use-chat-messages.ts) 通过 `entryType` 与少量 `metadata.subtype` 把日志重建为旧 `ChatMessage` 体系，属于 executor-era UI。
- [packages/shared/src/index.ts](/app/bkd/worktrees/n6ggd95o/l2801201/packages/shared/src/index.ts) 里的 `ChatMessage` 目前只有 `user / assistant / tool-group / task-plan / thinking / system / error`，没有 ACP-native 语义类型。
- [apps/frontend/src/components/issue-detail/SessionMessages.tsx](/app/bkd/worktrees/n6ggd95o/l2801201/apps/frontend/src/components/issue-detail/SessionMessages.tsx) 是旧消息体系的总渲染入口，`tool-group` 与 `task-plan` 都是在这里分发。
- ACP 后端实际上已经能产出更多协议语义：`plan`、`mode`、`tool_call`、`tool_call_update`、`diff`、`assistant chunk`、`turn completed`，但前端只能把这些数据继续压进旧消息模型。
- 这意味着当前 ACP UI 的主要问题不是“没有数据”，而是“数据语义已经存在，但前端展示模型过旧”。

## 实现结果

- 新增 `useAcpTimeline()`，为 ACP 日志构建独立时间线模型，不再复用旧的 `tool-group` 聚合逻辑。
- 新增 `AcpTimeline` 渲染组件，按协议顺序显示 assistant、plan、tool、session/system 条目。
- `SessionMessages` 现在根据 `engineType === 'acp'` 切换到 ACP 专用 renderer，非 ACP 引擎继续走原有链路。
- 现有工具 UI 组件被复用到 ACP renderer 中，因此 file diff、command output、search result 等已有展示能力都能直接继承。
