## 问题描述

打开 Issue 详情页后，"查看更多消息" 按钮有时不会显示，导致无法加载更早的历史聊天记录。

## 根因分析

**文件:** `apps/api/src/engines/issue/persistence/queries.ts`

SQL 层的 `entryType IN ('user-message', 'assistant-message', 'system-message')` 过滤不够精确。
大量不可见的 `system-message`（subtype 为 `init`、`hook_started`、`hook_response` 等）通过 SQL 过滤但被后续的 `isVisibleForMode()` JS 过滤器移除。

**问题链:**

1. SQL 查询使用 `LIMIT 61`（30 × 2 overfetch + 1）
2. SQL 返回 61 条，其中大量是 `system-message`（init/hook_started/hook_response 等不可见类型）
3. `isVisibleForMode()` JS 过滤后，可能只剩 < 31 条可见记录
4. 路由 `hasMore = filteredCount > 30` → `false`
5. 前端 `setHasOlderLogs(false)` → "查看更多" 按钮不显示
6. 但 DB 中实际还有更多可见记录（在 SQL LIMIT 之外）

**实测数据:** issue `ejdxh1b3` 旧 SQL 返回 11 条，JS 过滤后仅 5 条。55% 的 SQL 查询配额浪费在不可见记录上。

## 修复方案

将 `isVisibleForMode()` 的规则下沉到 SQL 层，使 SQL LIMIT 只计算真正可见的记录：

```sql
-- 旧：粗粒度类型过滤
entry_type IN ('user-message', 'assistant-message', 'system-message')

-- 新：精确匹配 isVisibleForMode() 规则
(
  (entry_type = 'user-message'
    AND (json_extract(metadata, '$.type') IS NULL
         OR json_extract(metadata, '$.type') != 'system'))
  OR entry_type = 'assistant-message'
  OR (entry_type = 'system-message'
    AND json_extract(metadata, '$.subtype') IN ('command_output', 'compact_boundary'))
)
```

同时移除路由中的 2x overfetch factor，改为标准 `+1` 检测 `hasMore`。
