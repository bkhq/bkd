# ENG-002 ACP SDK 接入并实现独立 ACP executor

- **status**: completed
- **priority**: P1
- **owner**: codex
- **plan**: PLAN-009
- **created**: 2026-03-13

## 描述

调查如何在 BKD 中使用 `@agentclientprotocol/sdk` 与代理通信，并将 ACP 从具体供应商实现中抽象为独立 engine。当前落地以 Gemini CLI 作为默认 ACP agent，但对外 `engineType` 不再绑定 `gemini`。

## 调查目标

- [x] 确认 `@agentclientprotocol/sdk` 的核心通信模型、传输层和最小 client API
- [x] 确认 `@zed-industries/claude-agent-acp` 的启动方式、依赖条件和支持的 Claude 能力
- [x] 盘点仓库现有 executor / protocol 抽象与 ACP 接入的差距
- [x] 给出最小 PoC 路径与实现边界

## 调查结果

- `@agentclientprotocol/sdk@0.16.1` 的最小 client 链路是 `ndJsonStream()` + `ClientSideConnection`，按 `initialize()` -> `newSession()` -> `prompt()` 建立 stdio ACP 会话。
- `@zed-industries/claude-agent-acp@0.21.0` 是一个独立 ACP agent，可直接用 `npx @zed-industries/claude-agent-acp` 启动；它内部依赖 `@anthropic-ai/claude-agent-sdk`，不是对当前仓库的 Claude stream-json 协议做薄包装。
- 本机未设置 `ANTHROPIC_API_KEY`，但 smoke test 中 `initialize` 与 `newSession` 成功，说明当前环境下该 agent 可以利用现有 Claude 本地登录态或 Claude Code 配置建立会话。
- smoke test 的 `prompt()` 请求未因 ACP 失败，而是返回上游账户错误 `"You've hit your limit · resets 3am (UTC)"`；这表明控制面链路已通，阻塞点在 Claude 账户额度。
- 仓库现状更接近两类模型：
  - `claude-code`: 每个 turn 启动独立进程，协议为 `stream-json`
  - `codex`: 长连接 / 会话型 agent，支持 `externalSessionId` 与运行中 follow-up
- 若引入 ACP，最自然的承载点是新增一个会话型 executor，复用 `externalSessionId`、`spawnFollowUp()`、`sendUserMessage()` 与现有流消费管线，而不是直接改写当前 Claude executor。
