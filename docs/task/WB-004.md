# WB-004 Whiteboard manual editing UX fixes

- **status**: completed
- **priority**: P1
- **owner**: claude
- **createdAt**: 2026-04-17

## Description

Audit of manual (non-AI) editing paths surfaced a set of real bugs and UX
papercuts. This task collects them into one pass:

1. **sortOrder collisions on create/reparent** — `onAddChild` and reparent both
   send no `sortOrder`, server falls back to DB default `'a0'`. Multiple
   siblings end up with identical sort keys → unstable order.
2. **Content edit triggers on any click** — markdown div has single-click
   handler; clicking a link or selecting text drops user into a textarea.
3. **Create has no optimistic update** — adding a child has ~300ms visible
   stall until server round-trip + query invalidation completes.
4. **New empty child doesn't auto-focus label edit** — every "+" is followed
   by a double-click to rename; should enter edit mode automatically.
5. **Drag snapback loses measured heights** — dropping into empty space calls
   `layoutMindmap` with unmeasured heights, producing a visible reflow flash.

## Acceptance Criteria

- [x] Adding N children in quick succession produces N distinct `sortOrder`
      values; siblings render in stable insertion order.
- [x] Reparenting a node computes a new `sortOrder` after the last sibling
      under the new parent.
- [x] Markdown content area enters edit mode on **double-click only**;
      single-click on links/text does not trigger edit mode.
- [x] Clicking "+" on a node inserts a placeholder immediately; the server
      response replaces the placeholder in-place.
- [x] A freshly created empty-label child auto-enters label edit mode with
      the input focused.
- [x] Dropping a dragged node onto empty space preserves previous measured
      positions (no reflow flash).

## Out of Scope

- Replacing `window.confirm` — consistent with rest of whiteboard, not part
  of this pass.
- Drag hover highlight / drop target feedback — tracked separately.
- i18n wording for cascade delete.
