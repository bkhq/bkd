# ENG-009 按功能模块拆分 ACP client

- **status**: completed
- **priority**: P1
- **owner**: codex
- **plan**: PLAN-018
- **created**: 2026-03-13

## 描述

当前 `apps/api/src/engines/executors/acp/acp-client.ts` 已超过 1000 行，同时承担协议事件类型、事件归一化、tool/assistant state、协议连接处理、spawn/query 模型查询等多类职责，维护成本偏高。

本任务目标是先完成职责拆分调查和模块化重构方案，确保后续重构以最小 import 变化完成。

## 调查结果

- `acp-client.ts` 当前约 1090 行，至少混合了 5 类职责：
  - ACP event / state 类型定义
  - stdio sink 与 child-process bridge
  - log normalizer（assistant/tool/result/plan/mode）
  - protocol handler（initialize / session / prompt / cancel）
  - 公共入口（`spawnAcpProcess()` / `queryAcpModels()`）
- 外部依赖面很小：
  - `executor.ts` 使用 `AcpLogNormalizer`、`normalizeAcpEvent`、`spawnAcpProcess`
  - `agents/base.ts` 使用 `queryAcpModels`
  - `test/acp-client.test.ts` 直接测 normalizer 导出
- 因此最稳的拆法不是直接改外部 import，而是先把内部模块拆开，再由 `acp-client.ts` 或 `index.ts` 继续统一 re-export。

## 实现结果

- 新增 `types.ts`，承载 ACP event / normalize state / tool state 类型。
- 新增 `transport.ts`，承载 event sink、child-process bridge 和 child spawn helper。
- 新增 `normalizer.ts`，承载 ACP log normalizer、assistant/tool/result/plan/mode 归一化逻辑。
- 新增 `protocol-handler.ts`，承载 ACP connection/session/prompt/cancel 逻辑。
- `acp-client.ts` 现在只保留公共入口与稳定导出：`normalizeAcpEvent`、`AcpLogNormalizer`、`spawnAcpProcess`、`queryAcpModels`。
